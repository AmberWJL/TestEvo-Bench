Abstract

In real software, tests and the production code they exercise are written and revised together; a change to a production method almost always arrives with a coordinated change to its test. Existing benchmarks for test generation and test update evaluate the two sides in isolation, which discards this co-evolution signal and makes it hard to tell whether a model actually understands how a change should propagate from implementation to test. We introduce LiveTestEvolBench, a live, continuously ingested benchmark of co-evolving test/production pairs mined from the public commit histories of real open-source Java projects. Each task is anchored to a concrete rev-pair in which a production edit is accompanied by a contemporaneous edit to the associated test suite, preserving the temporal and semantic link between the two artefacts. LiveTestEvolBench ships two evaluation tracks. In test update, a model must revise an existing test so that it still compiles, exercises the new behaviour of the modified production method, and passes — without weakening assertions or deleting coverage. In test generation, the model must write a brand-new test for newly added or updated production code that compiles, runs, passes, and meaningfully covers the changed lines. Because tasks are ingested from live commit streams rather than fixed at release time, LiveTestEvolBench enables contamination-aware evaluation: a time filter restricts tasks to any chosen rev2 date window, so results can be reported strictly against commits authored after a model's pre-training cutoff. We release the benchmark together with an open leaderboard that tracks compile pass rate, test pass rate, line and mutation coverage, test smell, and — for test update — the coverage delta on lines that were actually changed.

How to submit

  1. Fork the public livetestevolbench site repository.
  2. Run the provided evaluation harness on the tasks inside your chosen rev2 date window. For a fair contamination-aware report, pick a window that ends at or before your model's pre-training cutoff.
  3. Add a new entry to data/leaderboard.json under the relevant track, following the schema of the existing seeded rows (model name, organization, submission_date, paper_url, and all six metric fields).
  4. Open a pull request. Once the entry is reviewed and merged, the leaderboard re-deploys automatically.
